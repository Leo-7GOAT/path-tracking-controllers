# 简历项目描述建议

## 适配岗位

- 自动驾驶规控算法工程师
- 车辆控制算法工程师
- 运动规划 / 轨迹跟踪方向算法工程师

这个项目更偏**规控 / 控制算法**，和你截图里的“前沿规控算法开发工程师”匹配度更高。

## 一句话版本

搭建了一个自包含的轨迹跟踪控制 benchmark，统一实现并评测 Stanley、Pure PID、LQR 和有限时域预测控制器，在多类道路几何场景下对横向误差、完成时间和控制平滑性进行量化分析。

## 三条版简历描述

- 独立搭建车辆轨迹跟踪控制仿真平台，基于统一运动学自行车模型实现 `Stanley / Pure PID / LQR / Predictive Control` 四类控制器，并完成直线、曲线、圆弧三类典型场景对比实验。
- 自研纯 `NumPy` 三次样条路径生成与 benchmark 框架，统一终点制动、指标统计和可视化输出流程，自动生成轨迹图、指标面板、Markdown/CSV 结果表，提升实验复现性与工程完整度。
- 在曲线路径与持续曲率场景下完成精度-平滑性 trade-off 分析：`Stanley` 在几何跟踪精度上表现突出，`MPC-style` 预测控制在控制平滑性上更优，项目可直接支持后续扩展鲁棒控制或学习型控制器接入。

## 面试展开版

如果面试官追问，你可以这样展开：

1. 技术目标
   不是只“把控制器跑起来”，而是做成一个统一 benchmark，减少不同脚本、不同参数、不同终止条件带来的结论偏差。

2. 你做了什么
   重构成自包含仓库，补齐路径生成、车辆模型、图表导出、终点制动和统一指标统计；同时把各控制器统一到同一个接口下。

3. 你解决了什么问题
   原始脚本存在外部依赖缺失、环境门槛高、结果难复现的问题。现在仓库可以直接运行，并稳定生成可展示结果。

4. 你得出的结论
   `Stanley` 在几何路径跟踪上精度很高；`PID` 简单但在复杂曲率场景下容易引入更大误差和抖动；`LQR` 具备清晰状态反馈结构；`MPC-style` 预测控制在精度与控制平滑性之间更均衡。

5. 还能继续怎么做
   可以继续扩展控制噪声、参数失配、侧偏扰动、速度变化场景，或者进一步接入学习型控制器做对比。

## 项目标题建议

简历上建议不要只写“轨迹追踪控制器对比”，可以升级成下面这类标题：

- `自动驾驶轨迹跟踪控制器统一仿真与量化评测`
- `多控制器路径跟踪 benchmark 与控制性能分析`
- `车辆轨迹跟踪控制算法对比平台（PID / Stanley / LQR / MPC）`

## 更像岗位描述的写法

如果你想往“规控算法”岗位靠，可以写得更贴近 JD：

> 面向自动驾驶轨迹跟踪任务，完成多种经典规控算法原型实现、统一仿真验证与量化评测，分析不同控制策略在多曲率场景下的误差收敛特性与控制平滑性，为后续鲁棒控制与前沿规控算法迭代提供实验基线。
